# 白帽红帽分类逻辑

这份文档说明 `helmet_color_classify.py` 当前使用的安全帽颜色分类逻辑。整体流程是：先对视频逐帧运行 YOLO 姿态模型，检测出人员框和人体关键点；再根据头部关键点或双肩关键点估计每个人头盔所在的局部区域；然后将该区域转换到 HSV 颜色空间，分别计算白色像素比例和红色像素比例；最后按照当前代码里的二分类规则输出结果，其中红色像素占帽子分析区域 15% 及以上时判定为“工作人员”，否则统一判定为“管理人员”。

## 1. 整体处理流程

每一帧的处理顺序如下：

1. 使用 YOLO pose 模型检测人物框和 17 个关键点。
2. 对每个人单独估计头盔 ROI，也就是头盔的候选区域。
3. 对该 ROI 做颜色分析，计算 `white_ratio` 和 `red_ratio`。
4. 根据红色比例是否足够高，决定输出：
   - `工作人员`
   - `管理人员`
5. 将中文标签直接绘制回视频帧中，最后写入输出视频。

## 2. 头盔区域怎么取

代码不会直接拿整个人体框去判断颜色，而是先从人体关键点估计头盔区域，尽量把颜色分析限制在头部附近。

使用到的关键点主要有两组：

- 头部关键点：鼻子、左右眼、左右耳
- 肩部关键点：左肩、右肩

头盔 ROI 的估计规则如下：

- 如果头部关键点足够多且置信度达到阈值，就以头部关键点的加权中心作为头盔中心。
- 如果头部关键点不稳定，但左右肩可用，就以双肩中心向上推一个距离，估计头盔区域。
- 如果头部和肩部关键点都不可靠，就拿不到有效 ROI。

ROI 的大小不会固定死，而是跟人体框宽高、双肩间距一起动态计算。然后还会把 ROI 进一步限制在“人体框顶部附近”的一段区域里，避免把胸前反光背心、背景灯光或地面高亮区域误纳入头盔颜色判断。

## 3. 为什么要转成 HSV

代码不直接在 BGR 或 RGB 空间里比较颜色，而是先转成 HSV。原因是 HSV 更适合做这种规则判断：

- `H` 表示色相，适合区分红色这类有明显颜色倾向的目标。
- `S` 表示饱和度，适合区分“纯白/灰白”和“彩色”。
- `V` 表示明度，适合过滤太暗、无法可靠判断的区域。

在固定监控和夜间场景里，白色帽子通常表现为“亮度高、饱和度低”；红色帽子通常表现为“色相接近红色、饱和度高、亮度不能太低”。

## 4. 白色像素怎么定义

代码里会先计算白色掩码 `white_mask`。当前阈值是：

- `H: 0 ~ 179`
- `S <= 55`
- `V >= 170`

也就是说，只要像素足够亮、饱和度又足够低，就会被认为是“白色像素”。这里对白色不看具体色相，主要看“低饱和 + 高亮度”。

对应代码含义可以理解为：

- 颜色不能太艳
- 必须足够亮
- 满足这两个条件的像素，才会计入白色比例

## 5. 红色像素怎么定义

代码里会单独计算红色掩码 `red_mask`。当前阈值是：

- `S >= 75`
- `V >= 60`
- `H` 需要落在两段红色区间之一：
  - `0 ~ 18`
  - `145 ~ 179`

之所以分成两段，是因为 HSV 色环里红色刚好落在两端。也就是说，一个像素只有在“足够红、足够饱和、亮度也不是太低”时，才会被统计为红色像素。

## 6. 白色比例和红色比例怎么计算

在进入 HSV 判定前，代码还会做几步预处理，减少噪声：

- 只取头盔 ROI 的上部区域
- 左右各裁掉一部分边缘
- 做一次高斯模糊
- 对颜色掩码做 3x3 开运算，去掉小块噪点

之后会计算两个比例：

- `white_ratio = 白色像素数 / ROI 总像素数`
- `red_ratio = 红色像素数 / ROI 总像素数`

这两个比例本质上是在回答两个问题：

- 这个头盔区域里，有多少像素看起来像白色？
- 这个头盔区域里，有多少像素看起来像红色？

## 7. 当前代码里的最终判定规则

当前代码已经被改成强制二分类，不再输出“未知”。也就是说，它不是“白帽 / 红帽 / 未知”三分类，而是“红帽 / 非红帽”二分类。

当前生效的规则是：

- 如果 `red_ratio >= 0.15`，则判定为 `工作人员`
- 否则一律判定为 `管理人员`

换句话说，只要帽子分析区域里的红色像素达到 15%，就会被认定为红帽；白色比例不再参与最终分类否决。

## 8. 白色在当前版本里扮演什么角色

虽然当前版本最终只输出两类，但 `white_ratio` 仍然会被计算出来，用于调试观察和后续分析。它不再直接参与最终分类条件。当前代码更准确的说法是：

- 红色像素达到 15% -> 工作人员
- 其余所有情况 -> 管理人员

## 9. 这套规则的优点和限制

优点：

- 不需要重新训练模型
- 对固定机位、固定场景比较容易落地
- 规则可解释，方便后续调阈值

限制：

- 当前版本不再输出“未知”，所以灰帽、黄帽、黑帽、颜色不清晰、ROI 不准等情况，都会被归到“管理人员”
- 夜间反光、强灯、高亮背景、背心反光都可能干扰白色比例
- 如果头部关键点不稳，头盔 ROI 可能偏移，进而影响颜色判断

## 10. 如果以后要恢复“白帽 / 红帽 / 未知”三分类

可以把最终规则重新改回三分类思路：

- 红色比例足够高且占优 -> `工作人员`
- 白色比例足够高且占优 -> `管理人员`
- 两者都不明显，或关键点不可靠 -> `未知`

这样会更保守，也更符合“白帽 / 红帽”显式分类的语义，但代价是输出结果里会重新出现一部分“未知”。
