# 人体框与安全帽框匹配分类方案

本文档给出一个替代当前“根据人体关键点估计帽子 ROI 再做颜色分析”的方案。考虑到当前视频清晰度有限，头部小目标不稳定、夜间模糊明显，因此本阶段不引入 `head detector`，而是先做一个更轻、更稳、更容易验证收益的最小闭环：

- 人体框 `person bbox`
- 安全帽框 `helmet bbox`

然后基于“人框上半区 + 几何约束 + 一对一分配”完成帽子归属，最后仅在 `helmet bbox` 内做颜色分类。这个方案的核心目标是：在不增加额外检测链路复杂度的前提下，先解决当前帽子 ROI 偏大、偏低、容易把背景和脸部一起框进去的问题。

## 1. 方案目标

当前实现的主要问题是：

- 帽子区域是估计出来的，不是真实检测框
- ROI 容易包含背景高亮区域、脸部、脖子和头发
- 一旦 ROI 偏移，后续红白像素占比就会明显失真
- 现有方案对关键点稳定性依赖较高，不适合当前夜间视频

新方案本阶段要达到的目标是：

- 用独立的安全帽检测框替代估计 ROI
- 只在帽子框内做颜色统计，减少背景干扰
- 不引入 `head detector`，先控制系统复杂度
- 在多人场景下尽量避免把别人的帽子匹配到当前人身上
- 输出清晰的结构化结果，便于后续调试和评估

## 2. 总体流程

每一帧的处理流程建议如下：

1. 检测所有人，得到 `person bbox`
2. 检测所有安全帽，得到 `helmet bbox`
3. 为每个 `person bbox` 生成“上半区匹配区”
4. 计算 `helmet -> person` 候选匹配关系
5. 通过几何约束过滤不合理候选
6. 对剩余候选做一对一分配
7. 匹配成功：
   - 判定该人“戴安全帽”
   - 在对应 `helmet bbox` 内做颜色分类
8. 匹配失败：
   - 判定该人“未匹配到安全帽”

输出结果建议分为三层：

- 第一层：是否成功匹配到帽子
- 第二层：帽子颜色类别，例如红帽、白帽、黄帽
- 第三层：业务身份映射，例如管理人员、工作人员

说明：

- “未匹配到安全帽”不等价于“确认未戴帽”
- 在当前阶段，产品侧应保留 `unknown / 无法判断` 状态，避免把漏检直接打成违规

## 3. 为什么不用“人与帽子框有重合”直接判定

如果直接用“`person bbox` 和 `helmet bbox` 有重合”作为规则，误匹配会比较多，原因是：

- 人框本身很大，帽子只要压到人框边缘一点就可能被算进去
- 多人靠近时，一个帽子可能同时与两个人人框重合
- 在拥挤场景下，旁边人的帽子很容易被错配

因此推荐的规则不是“人与帽子框有重合”，而是：

- 帽子框中心点落在该人的上半区匹配区内或附近
- 帽子框主要位于该人的上半区
- 帽子框大小、位置与该人的人框尺度关系合理

## 4. 推荐的匹配规则

### 4.1 人框上半区匹配区生成

本方案不使用 `head bbox`，而是直接从 `person bbox` 构造一个稳定、可配置的“上半区匹配区”作为帽子锚定区域。

设人体框为：

- `px1, py1, px2, py2`
- `person_w = px2 - px1`
- `person_h = py2 - py1`

建议先生成一个保守的帽子候选匹配区：

- `match_x1 = px1 + 0.22 * person_w`
- `match_x2 = px2 - 0.22 * person_w`
- `match_y1 = py1 - 0.06 * person_h`
- `match_y2 = py1 + 0.22 * person_h`

这样做的目的：

- 横向上避免吃到手臂、旁边人和背景
- 纵向上允许帽子略高于人框顶部
- 不再退回整个人框做匹配，减少误匹配面积

以上比例不是固定真值，而是当前机位的初始默认值，后续应作为配置参数调优。

### 4.2 `helmet -> person` 几何匹配条件

对于每个 `helmet bbox`，计算其与每个 `person bbox` 上半区匹配区的关系，推荐使用以下组合规则：

- 条件 1：`helmet center` 落在 `match_x1 ~ match_x2` 范围内，或水平偏差足够小
- 条件 2：`helmet bbox` 至少有一部分位于 `match_y1 ~ match_y2` 范围内
- 条件 3：`helmet bbox` 的底边不能明显低于 `py1 + 0.30 * person_h`
- 条件 4：`helmet bbox` 宽度与 `person_w` 的比例在合理范围内，例如 `0.08 ~ 0.38`
- 条件 5：`helmet bbox` 高度与 `person_h` 的比例在合理范围内，例如 `0.05 ~ 0.22`

一个简单可落地的判定可以写成：

- `abs(helmet_cx - person_cx) <= 0.28 * person_w`
- `helmet_y2 >= match_y1`
- `helmet_y1 <= match_y2`
- `helmet_y2 <= py1 + 0.30 * person_h`
- `0.08 <= helmet_w / person_w <= 0.38`
- `0.05 <= helmet_h / person_h <= 0.22`

满足以上条件后，再按距离、重叠比例和尺度合理性打分。

### 4.3 多人多帽的一对一匹配

当一个帽子可能匹配多个人，或一个人附近有多个帽子时，需要做一对一分配。推荐方法：

- 先计算所有候选匹配对的分数
- 分数可由以下项组合：
  - 帽子中心点到人框中心上部参考点的距离
  - 帽子框与上半区匹配区的重叠比例
  - 帽子框是否主要位于人框顶部区域
  - 帽子框尺度是否合理
- 按分数从高到低排序
- 使用贪心策略依次分配，已被占用的人和帽子不再重复匹配

如果后续场景更复杂，也可以换成匈牙利算法做全局最优分配。但在当前固定机位场景下，建议先用贪心策略验证收益。

## 5. 帽子颜色分类建议

新方案下，颜色分析不再使用估计头部 ROI，而是直接对 `helmet bbox` 做分析。这样可以显著减少背景混入。

颜色分类建议如下：

1. 从 `helmet bbox` 中再裁一个更保守的内部区域
2. 只分析中间 `70% ~ 80%` 区域，去掉边缘
3. 转到 HSV 或 Lab 空间
4. 计算红、白、黄等颜色的掩码
5. 使用最大连通域或主色占比做最终判定

建议不要直接用整个检测框面积作为分母，而应优先使用：

- 帽子主体掩码面积
- 或颜色连通域面积

这样可以进一步降低背景、反光和边缘噪声的影响。

## 6. 推荐的数据结构

建议每个人输出一条结构化结果，示例字段如下：

```text
frame_person_index
person_box
person_match_zone
helmet_status
helmet_box
helmet_match_score
helmet_color
helmet_color_score
role_label
role_confidence
decision_reason
```

其中：

- `frame_person_index` 表示当前帧内的人体序号，不表示稳定跨帧 ID
- `person_match_zone` 保存该人的上半区匹配区
- `helmet_status` 建议取值：`matched`、`unmatched`、`unknown`
- `helmet_box` 保存最终归属到该人的安全帽框
- `helmet_color` 保存颜色分类结果
- `role_label` 保存业务身份映射结果，例如管理人员、工作人员
- `decision_reason` 用于记录“为什么这么判”，方便后续排错

## 7. 这个方案的优势

相比“姿态点估计帽子 ROI”的方案，这个方案的优点更直接：

- 帽子框来自检测器，定位通常更准
- 颜色分析只在帽子框内部进行，背景污染更少
- 不新增头部检测链路，工程复杂度更低
- 更适合作为最小闭环方案快速验证收益
- 是否匹配到帽子和帽子颜色两个问题被拆开，逻辑更清晰
- 更适合扩展到红帽、白帽、黄帽、蓝帽等多类别

## 8. 这个方案的风险和前提

该方案也有明显前提，需要提前说明：

- 必须有可用的安全帽检测模型
- 远距离小目标时，帽子检测可能漏检
- 夜间和模糊场景下，帽子框可能不稳定
- 遮挡严重时，帽子可能和人框错配
- 只靠 `person + helmet` 的几何关系，在多人重叠时仍然可能存在误匹配

因此，该方案能否比当前方案更稳，关键取决于 `helmet detector` 的质量，尤其是：

- 小目标检测能力
- 夜间和模糊场景下的召回率
- 红白黄等不同颜色安全帽的检测一致性
- 多人靠近时的稳定性

## 9. 推荐实施顺序

建议分三步落地，而不是一次性全部重做：

### 第一步：先接入 `person + helmet`

- 保留现有人检测
- 新增安全帽检测
- 生成人框上半区匹配区
- 输出“人框 + 帽子框 + 匹配关系”
- 暂时先不改颜色分类逻辑

先验证两个问题：

- 帽子能不能稳定检出
- 帽子能不能稳定匹配到正确的人

### 第二步：把颜色分析切换到帽子框

- 不再使用当前估计头部 ROI 做颜色判断
- 改为直接在 `helmet bbox` 内统计颜色
- 加入中心区域裁剪和连通域过滤

### 第三步：加时序稳定

对视频场景，建议加入短时跟踪或时序平滑：

- 同一人的帽子框做滑动平均
- 连续多帧匹配成功后再确认“戴帽”
- 颜色结果做 3 到 5 帧投票

这样可以减少单帧抖动和偶发误检。

## 10. 验收建议

建议在当前机位下先标注一批评估样本，再比较“现方案”和“`person + helmet` 方案”的收益。核心指标建议至少包括：

- 帽人匹配准确率
- 戴帽召回率
- 未匹配误报率
- 颜色分类准确率
- `unknown / 无法判断` 占比
- 单帧耗时与整体 FPS

只有当 `person + helmet` 明显优于现方案，且满足性能要求，再考虑是否有必要进入下一阶段复杂化。

## 11. 推荐结论

推荐把当前方案先升级为：

- `person detector` 检测人
- `helmet detector` 检测安全帽
- 基于“人框上半区 + 几何约束”做 `helmet -> person` 匹配
- 在 `helmet bbox` 内完成颜色分类

这个方案比“直接看人框和帽子框是否重合”更合理，也比当前“用人体关键点估计帽子 ROI”更稳。更重要的是，它避免了一开始就引入 `head detector` 带来的额外复杂度，适合作为当前阶段最值得投入的最小闭环方案。
